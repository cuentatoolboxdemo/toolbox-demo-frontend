# Phase 9 Plan 1: Login Gate and Role Router

## Objectives
Introduce an authentication barrier between the tenant directory (`/`) and the tenant-specific applications (`/[tenant]` and `/admin/[tenant]`). The login credential provided will determine the user's role and their ultimate destination.

## Implementation Steps

1. **Update Auth API (`/api/auth/login/route.ts`)**:
   - Currently, it only validates "demo123" for admin logic.
   - Refactor to accept an additional hardcoded regular user password (e.g., "user123").
   - Update the `/api/auth/login` response to return the authenticated `role` (either `admin` or `user`).
   - Save session data encoding the discovered role.

2. **Create `LoginDialog` Component**:
   - Create `src/components/auth/LoginDialog.tsx` containing the password input form (extracted/adapted from the current Admin panel).
   - This component should be a modal (Dialog) from Shadcn UI or simply take over the screen.

3. **Modify the Directory Hub (`src/app/page.tsx`)**:
   - Currently, the tenant cards are hard links `<Link href={\`/${tenant.id}\`}>`.
   - Prevent standard navigation. Capture the click event to open `LoginDialog` instead, tracking which `tenant` was clicked.

4. **Implement Routing Logic**:
   - Inside `LoginDialog`, upon successful authentication, read the returned `role`.
   - `if (role === 'admin') router.push('/admin/[tenant]');`
   - `else router.push('/[tenant]');`

5. **Clean up isolated Admin Auth**:
   - The current `/admin/[tenant]` views require you to log in *again* if not authenticated. We can leave it as a fallback route guard, but if the user arrives already authed from `/`, they should see the panel directly without re-logging in.

## Verification
- Run `npm run lint` and ensure no TypeScript typing errors or unused imports.
- Click a tenant in the directory `/` and verify the password modal spawns.
- Type `user123` -> verify it lands on the AI Chat.
- Go back to `/`. Click a tenant -> type `demo123` -> verify it lands on the Admin view.
