<objective>
Wire the two external webhook URLs (chat inference and n8n document ingest) as environment variables so they can be changed without touching source code.

Purpose: Phase 2 (chat) and Phase 3 (admin upload) both POST to external webhooks. These URLs are TBD and will change. The app must read them from environment variables, not hardcode them. This plan creates the `.env.local` file and documents the pattern.
Output: `.env.local` with placeholder webhook URLs, `.env.example` as the template, and `.gitignore` updated to exclude secrets.
</objective>

<task type="auto">
  <name>Task 1: Create .env.local and .env.example with webhook URL variables</name>
  <files>.env.local, .env.example, .gitignore</files>
  <action>
    1. Create `.env.example` (safe to commit — contains only placeholder values):
    ```
    # Toolbox AI — Environment Variables
    # Copy this file to .env.local and fill in your actual values.
    # NEVER commit .env.local to source control.

    # External webhook URL for chat inference (receives { question, tenant, history, systemPrompt })
    NEXT_PUBLIC_CHAT_WEBHOOK_URL=https://your-n8n-instance.com/webhook/chat

    # External webhook URL for document ingest (receives FormData with PDF file)
    NEXT_PUBLIC_INGEST_WEBHOOK_URL=https://your-n8n-instance.com/webhook/ingest
    ```

    2. Create `.env.local` (never committed — contains placeholder values for local dev):
    ```
    # Local development environment variables.
    # Replace placeholder values with real webhook URLs when available.

    NEXT_PUBLIC_CHAT_WEBHOOK_URL=https://placeholder.example.com/webhook/chat
    NEXT_PUBLIC_INGEST_WEBHOOK_URL=https://placeholder.example.com/webhook/ingest
    ```

    3. Check `.gitignore` — if it already exists (from create-next-app scaffold), verify `.env.local` is listed. If it is missing, add the following lines:
    ```
    # Environment variables
    .env.local
    .env.*.local
    ```

    Variable naming rationale:
    - Using `NEXT_PUBLIC_` prefix because both chat (Phase 2) and admin upload (Phase 3) call these webhooks from the browser (client-side fetch). Without `NEXT_PUBLIC_`, the variables would only be available server-side (API routes / Server Components).
    - If the team later moves webhook calls to Server Actions or API routes, the `NEXT_PUBLIC_` prefix can be dropped, but for this demo the simpler client-side fetch approach is used.
  </action>
  <verify>
    1. `cat .env.example` — shows both NEXT_PUBLIC_CHAT_WEBHOOK_URL and NEXT_PUBLIC_INGEST_WEBHOOK_URL with placeholder values.
    2. `cat .env.local` — shows both variables with placeholder values.
    3. `cat .gitignore` — contains `.env.local` (and/or `.env*.local`).
    4. `npm run dev` still starts without errors (env vars load silently even with placeholder values).
    5. In a quick test, add `console.log(process.env.NEXT_PUBLIC_CHAT_WEBHOOK_URL)` to `src/app/page.tsx`, run dev, check browser console — the placeholder URL string appears (proves Next.js injects NEXT_PUBLIC_ vars into client bundles). Remove the console.log after verifying.
  </verify>
  <done>
    .env.local exists with both webhook URL variables set to placeholders. .env.example exists as the committed template. .gitignore excludes .env.local. `npm run dev` starts cleanly. NEXT_PUBLIC_ variables are accessible in browser console (confirmed via test log).
  </done>
