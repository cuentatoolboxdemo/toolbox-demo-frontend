# Phase 11 Plan 1: Dynamic Tenant Theming & Avatar Customization

## Objectives
Introduce isolated visual branding per tenant. "Calzedonia" needs to feel like Calzedonia (e.g. monochromatic or distinct colors, possibly its logo), while "Sabores" should have its own palette. This enforces psychological tenant separation visually and accomplishes the UX polishing.

## Implementation Steps

1. **Extend Tenant Definitions (`src/lib/tenants.ts`)**:
   - Update the `Tenant` interface to accommodate a `theme` object.
   - The `theme` object should define color classes (or raw hex codes) matching Tailwind arbitrary attributes for things like: `primary`, `glow`, `ringActive`, and an optional `logoUrl`.
   - Update `TENANTS` dictionary configuring Calzedonia (e.g., black/white/gray elegant scheme), Sabores (e.g., orange/red warm scheme), and IDE Marketing (e.g., tech blue/indigo).

2. **Refactor `AnimatedAvatar` for Dynamic Styling**:
   - Update the properties definition to accept `theme` configurations.
   - Replace the hardcoded `bg-blue-X`, `border-indigo-X`, and `shadow` colors with runtime style injections or computed Tailwind classes based on the `theme` prop.
   - If `theme.logoUrl` is provided, replace the central generic "Pulse Eye" with an `<img>` tag that retains the animation states (scale, pulsing). 
   - If `theme.logoUrl` is omitted, retain the glowing inner circle but colorize it using `theme.primary`.

3. **Propagate Context through `ChatInterface`**:
   - Read the styling instructions from the isolated `tenant` object provided to `ChatInterface`.
   - Inject those properties correctly into `AnimatedAvatar`, and consider making the text bubbles adopt the tenant's primary UI color instead of strict default gray/blue.

## Verification
- Boot `/calzedonia` and ensure the avatar/colors align with the configured elegant/dark theme, pulling the logo if mocked.
- Boot `/sabores` and confirm warm colors instantly display upon render.
- Test fallback behavior for missing configurations.
